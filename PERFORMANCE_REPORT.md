# QZSS災危通報 統合防災地図 — 性能改善・品質検証レポート

対象システム: みちびき(QZSS)から受信した災危通報(地震・津波・Jアラート・
Lアラート・気象警報等)を、ラズベリーパイでデコードして地図に表示する
統合防災サイネージシステム。

本レポートは、実測に基づく高速化と、大量のテストケースを使った
品質検証で見つかった不具合の修正記録をまとめたもの。

---

## 1. 受信〜サーバー送信の高速化

### 1.1 まず実測でボトルネックを特定した

「なんとなく速くする」のではなく、各処理ステップの所要時間を先に実測した。

| 処理 | 実測時間 |
|---|---|
| デコード(azarashi) + JSON変換 + 本文生成 | **0.07〜0.10 ms** |
| HTTP送信(接続を毎回新規に張る場合) | **平均 161.7 ms**(最小147ms/最大231ms) |
| HTTP送信(TCP/TLS接続を使い回す場合) | **平均 90〜96 ms**(初回除く平均89.5ms) |

**結論**: デコード処理はボトルネックではなく(1000倍以上の差)、
サーバーへのHTTP送信がほぼ全てを占めていた。特にTLSハンドシェイクの
コストが大きく、接続を使い回すだけで**約40%短縮**できることを確認した。

### 1.2 実施した改善

1. **送信の非同期化(キュー+専用ワーカースレッド)**
   受信ループが `urllib.request.urlopen()` を直接呼んでブロックしていた
   構成をやめ、キューに積むだけ(実測 **0.192 ms**)で次のシリアルバイト
   読み取りに戻れるようにした。通報が連続して届いても、送信待ちで
   受信が止まらない。

2. **HTTP接続のkeep-alive化**
   `urllib.request` → `http.client.HTTPSConnection` の使い回しに変更。
   切断時は自動で1回だけ再接続する。

3. **送信失敗時の自動リトライ**
   実際の警報は最大5回・3秒間隔で自動再送。ハートビート(死活監視、
   30秒おき)は次が来るためリトライせず即座に諦め、キューが古い1件で
   詰まらないようにした。

4. **シリアル読み取りループの微修正**
   バイト連結を `bytes`(毎回新規オブジェクト生成)から `bytearray`
   (in-place追記)に変更。

### 1.3 結果

| 指標 | 改善前 | 改善後 |
|---|---|---|
| 受信ループが送信待ちで止まる時間 | 約150〜230 ms | **約0.2 ms**(約1000倍) |
| 実際の送信完了までの時間 | 約150〜230 ms | **約90〜100 ms**(約40%短縮) |
| 送信失敗時の挙動 | その場で諦めてログのみ | 自動リトライ(最大5回) |

## 2. 地図描画の軽量化(ラズパイ3等の非力な端末向け)

### 2.1 段階的ロード

起動時に全データを一括で待ってから描画する構成をやめ、優先度順に
3段階へ分割した。

- **段階0**: `style.json` のみ(地図の生成に必須なものだけを待つ)
- **段階1(優先)**: 警報エリアの描画に直結するデータ
  (都道府県・津波沿岸・気象警報区域)を並行取得し、揃い次第すぐ描画
- **段階2(背景読み込み)**: 使用頻度の低いデータ
  (震央地名ルックアップ、市区町村ポリゴン)は地図表示後にバックグラウンドで読み込み、
  初回描画の完了を待たせない

実測(ローカル環境):

| データ | 取得完了タイミング |
|---|---|
| style.json | +203ms |
| 下地地図(pmtiles) | +368ms |
| **都道府県・津波・気象警報(優先グループ)** | **+492ms** |
| 震央地名ルックアップ(背景) | +539ms |
| 市区町村ポリゴン(背景) | +607ms |

### 2.2 市区町村ポリゴンの遅延読み込み

Lアラートの市区町村単位描画に使う国土数値情報ポリゴン(約1900件、
全レイヤー中もっとも重い)は、起動時ではなく地図表示後にバックグラウンドで
読み込む。読み込み完了前にLアラートが届いた場合も、完了後に自動で
塗りを反映する(取りこぼしなし)。

### 2.3 無駄な描画の削減

`fallback-landmass`(タイル未提供のズームレベル用の簡易地図)に
`maxzoom: 5` を追加。本物のベクトルタイルの下に完全に隠れて見えなく
なった後も描画コストを払い続けていた無駄を削減した。

### 2.4 データ精度とファイルサイズの両立

Lアラートの市区町村表示で、当初使っていた簡略化データ(0.1%簡略化、
445KB)は小さな区の形が崩れて認識しづらかったため、より精度の高い版
(1%簡略化、3.8MB)に差し替え。ファイルサイズ増加分はgzip圧縮
(転送量 約1.2MB)とサーバー側の`compression`ミドルウェア導入で吸収した。

## 3. 品質検証: 52パターンのテストケースで発見・修正した不具合

`test/replay_test_cases.py` で、地震・津波・Jアラート・Lアラート(楕円/
市区町村)・気象警報・南海トラフ・火山・降灰・洪水・訓練放送・
デコードエラー等、**52パターン**のテストケースを作成し、ローカル環境で
連続再生して目視検証した。

### 発見した重大な不具合

**サーバー側で、取消・解除信号を1件受信するたびに、保持していた
「現在アクティブな通報」を全件消去していた**(`activeReports = []`)。

これにより、例えば緊急地震速報の取消が1件届いただけで、同時に
アクティブだった無関係の気象警報やLアラートまで、新規接続した
ブラウザから見て消えてしまう状態だった。テストケースを流したところ、
52件中ほぼ全てが最終的に1件しか画面に残らないという明らかな
異常で発覚した。

**修正**: 取消・解除の対象を「災害種別+対象地域」等で判定する
グルーピングキーを実装し、一致するものだけを取り除くよう変更。
判定できない場合は何も消さない(安全側に倒す)設計にした。

修正後、52件中カード表示すべき33件(取消・解除された19件を除く)が
正しく残ることを確認。

### 副次的に発見・修正した不具合

- **Lアラートの解除(All Clear)がサーバー側の「終了信号」判定に
  含まれておらず**、解除後も新規接続したブラウザに再送され続けていた
- **Jアラートに解除処理が一切実装されておらず**、解除メッセージが
  通常のアラートとして積み上がってしまう構成だった
  (Lアラートと同様の仕組みを追加して解消)

## 5. ラズパイ実機キオスクの高速化・安定化(2026-07-16〜17)

「ラズパイ3でも快適に動く」を実際の実機検証で裏付けるため、GPU・
表示サーバー・OS・地図データ・描画処理のすべてを見直した。

### 5.1 GPUを完全に無効化し、レンダラークラッシュを根絶

実機で20〜25分間隔の再起動(`--type=renderer`のクラッシュダンプ)が
発生していた。クラッシュダンプの中身を確認したところ、GPU-GL
コンテキスト関連のクラッシュだった。

原因を切り分けたところ、**Pi 3B+のGPUドライバ(VideoCore IV +
Mesa)がES3.0のEGLコンテキスト生成に必ず失敗する**
(`eglCreateContext ES 3.0 failed with error EGL_BAD_ATTRIBUTE`)ことが
判明。X11・Wayland両方、GPUを部分的に使う設定も含めて何通りも試したが
すべて同じ失敗だった。

**対策**: `--use-angle=swiftshader --enable-unsafe-swiftshader`
(ソフトウェアWebGL)に統一。単純な`--disable-gpu`はWebGL自体が
初期化に失敗する(MapLibre GL JSはWebGL必須でフォールバックが無い)
ため使えないこともあわせて確認した。

**結果**: クラッシュが発生しなくなった。

### 5.2 表示サーバーをWayland(labwc)からX11(openbox)へ切り替え

画面合成(コンポジタ)処理だけでCPUを常時69%前後消費していることが
判明。ソフトウェアWebGL化でCPU負荷が上がった分、ここが特に効いていた。

`/etc/lightdm/lightdm.conf`の`user-session`/`autologin-session`を
`rpd-labwc`→`rpd-x`(Raspberry Pi OSに元から用意されている、
openbox+lxsessionベースのX11セッション)に変更。

**結果**: コンポジタのCPU使用率が69%前後→ほぼ0%に低下。このセッション
最大の体感速度改善。

### 5.3 DPMS(画面電源管理)を無効化

マウス・キーボードが接続されていないキオスクは無操作状態が続くため、
X11標準のDPMSにより無操作10分で画面が真っ暗になる不具合があった。
`xset s off; xset s noblank; xset -dpms`をautostartに追加して解消。

### 5.4 CPU優先度(Nice)の最適化

警報の受信・描画に関わるプロセスと、それ以外の定期メンテナンス処理を
明確に分離した。

| サービス | 役割 | Nice値 |
|---|---|---|
| qzss-decoder | 受信機からのデコード・送信 | -5(優先) |
| qzss-map | 地図アプリ本体(WebSocket配信) | -5(優先) |
| qzss-kiosk | Chromiumキオスク表示 | -5(優先) |
| qzss-cloud-health-check / qzss-kiosk-watchdog / qzss-report-status / qzss-update-check / qzss-urgent-check | 定期監視・OTA確認等 | 10(低優先) |

CPU競合が発生しても、警報の受信・描画側が常にスケジューラに優先される。

### 5.5 キオスク起動時間の短縮

Xセッションが立ち上がるまでの待ちを、固定5秒の`sleep`から
「xrandrが実際の解像度を報告するまで」ポーリングする方式に変更
(最大10秒、0.2秒間隔)。準備が早く整った場合は5秒未満で起動できる。

(単純にXソケットの有無だけを見る初期実装では、解像度ネゴシエーション
完了前にChromiumが起動し、意図しない低解像度で表示される不具合が
実機で発生したため、xrandrで実際のモードを確認する方式に修正した)

### 5.6 地図データの軽量化

- 下地地図(pmtiles)をズーム4〜7のみに限定(176KB)
- どこからも参照されていなかった`base_slim_simplified.pmtiles`
  (161KB)を削除
- キオスク専用の市区町村データを25%簡略化(1.0MB、形状が崩れない
  範囲で公開用の40%よりやや簡略化)
- 最大ズームレベルを13→9に変更(対象地域の把握にそれ以上の拡大は
  不要と実機・実際の操作感で確認。キオスク・公開Web版とも統一)

### 5.7 描画ライブラリのローカル化(真のオフライン対応)

`maplibre-gl.js`・`pmtiles.js`をunpkg CDNから読み込んでいたが、
キオスクは`--incognito`起動でキャッシュが毎回消えるため、
**電波が無い状態で再起動すると地図自体が表示できない**構成になって
いた。両ライブラリを`public/vendor/`にローカル同梱し、真にオフラインで
動作するようにした。

### 5.8 描画の優先順位付け

警報受信時の処理順を「警報エリアの塗りつぶし(最優先)→震源マーカー・
震度バッジ→カメラズーム」の順に統一。安全に関わる情報(対象範囲の色)が
必ず最初に画面へ反映されるようにした。

### 5.9 エンドツーエンドのレイテンシ計測基盤

信号受信からブラウザでの描画完了までを5段階のタイムスタンプで計測:

```
T0 信号受信(受信機、Python)
T1 デコード完了(受信機、Python)
T2 サーバー受信(Node.js /ingest)
T3 WebSocket配信直前(Node.js)
T4 描画完了(ブラウザ、塗りつぶし反映直後)
```

1件の通報につき1行のログに各区間の所要時間(T1-T0, T2-T1, T3-T2,
T4-T3, 合計)がまとまって出力される(Cloud Run/journalctlログ)。
今後、実際の受信データで「どこが遅いか」を数値で追えるようになる。

### 5.10 まとめ

| 項目 | 変更前 | 変更後 |
|---|---|---|
| 画面合成のCPU使用率 | 常時69%前後(Wayland) | ほぼ0%(X11) |
| GPU起因のレンダラークラッシュ | 20〜25分間隔 | 発生しなくなった |
| 下地地図pmtilesのサイズ | ズーム制限なし | 176KB(ズーム4〜7) |
| 最大ズームレベル | 13 | 9(キオスク・公開Web版とも) |
| オフライン動作 | CDN依存で実質不可 | ローカル同梱で確実に動作 |
| キオスク起動の待ち時間 | 固定5秒 | 実際の準備完了次第で短縮 |
| 警報描画時のCPU優先度 | 他処理と同列 | 常に優先(Nice=-5 vs 10) |
| レイテンシの可視化 | 体感のみ | T0〜T4の5段階を実測ログ化 |

CPU使用率とクラッシュ頻度は実測値。レイテンシの絶対値(何ms→何ms)は
計測基盤を入れたばかりで、実際の受信データが蓄積され次第、本レポートに
追記する。
