# QZSS災危通報 統合防災地図 — 性能改善・品質検証レポート

対象システム: みちびき(QZSS)から受信した災危通報(地震・津波・Jアラート・
Lアラート・気象警報等)を、ラズベリーパイでデコードして地図に表示する
統合防災サイネージシステム。

本レポートは、実測に基づく高速化と、大量のテストケースを使った
品質検証で見つかった不具合の修正記録をまとめたもの。

---

## 1. 受信〜サーバー送信の高速化

### 1.1 まず実測でボトルネックを特定した

「なんとなく速くする」のではなく、各処理ステップの所要時間を先に実測した。

| 処理 | 実測時間 |
|---|---|
| デコード(azarashi) + JSON変換 + 本文生成 | **0.07〜0.10 ms** |
| HTTP送信(接続を毎回新規に張る場合) | **平均 161.7 ms**(最小147ms/最大231ms) |
| HTTP送信(TCP/TLS接続を使い回す場合) | **平均 90〜96 ms**(初回除く平均89.5ms) |

**結論**: デコード処理はボトルネックではなく(1000倍以上の差)、
サーバーへのHTTP送信がほぼ全てを占めていた。特にTLSハンドシェイクの
コストが大きく、接続を使い回すだけで**約40%短縮**できることを確認した。

### 1.2 実施した改善

1. **送信の非同期化(キュー+専用ワーカースレッド)**
   受信ループが `urllib.request.urlopen()` を直接呼んでブロックしていた
   構成をやめ、キューに積むだけ(実測 **0.192 ms**)で次のシリアルバイト
   読み取りに戻れるようにした。通報が連続して届いても、送信待ちで
   受信が止まらない。

2. **HTTP接続のkeep-alive化**
   `urllib.request` → `http.client.HTTPSConnection` の使い回しに変更。
   切断時は自動で1回だけ再接続する。

3. **送信失敗時の自動リトライ**
   実際の警報は最大5回・3秒間隔で自動再送。ハートビート(死活監視、
   30秒おき)は次が来るためリトライせず即座に諦め、キューが古い1件で
   詰まらないようにした。

4. **シリアル読み取りループの微修正**
   バイト連結を `bytes`(毎回新規オブジェクト生成)から `bytearray`
   (in-place追記)に変更。

### 1.3 結果

| 指標 | 改善前 | 改善後 |
|---|---|---|
| 受信ループが送信待ちで止まる時間 | 約150〜230 ms | **約0.2 ms**(約1000倍) |
| 実際の送信完了までの時間 | 約150〜230 ms | **約90〜100 ms**(約40%短縮) |
| 送信失敗時の挙動 | その場で諦めてログのみ | 自動リトライ(最大5回) |

## 2. 地図描画の軽量化(ラズパイ3等の非力な端末向け)

### 2.1 段階的ロード

起動時に全データを一括で待ってから描画する構成をやめ、優先度順に
3段階へ分割した。

- **段階0**: `style.json` のみ(地図の生成に必須なものだけを待つ)
- **段階1(優先)**: 警報エリアの描画に直結するデータ
  (都道府県・津波沿岸・気象警報区域)を並行取得し、揃い次第すぐ描画
- **段階2(背景読み込み)**: 使用頻度の低いデータ
  (震央地名ルックアップ、市区町村ポリゴン)は地図表示後にバックグラウンドで読み込み、
  初回描画の完了を待たせない

実測(ローカル環境):

| データ | 取得完了タイミング |
|---|---|
| style.json | +203ms |
| 下地地図(pmtiles) | +368ms |
| **都道府県・津波・気象警報(優先グループ)** | **+492ms** |
| 震央地名ルックアップ(背景) | +539ms |
| 市区町村ポリゴン(背景) | +607ms |

### 2.2 市区町村ポリゴンの遅延読み込み

Lアラートの市区町村単位描画に使う国土数値情報ポリゴン(約1900件、
全レイヤー中もっとも重い)は、起動時ではなく地図表示後にバックグラウンドで
読み込む。読み込み完了前にLアラートが届いた場合も、完了後に自動で
塗りを反映する(取りこぼしなし)。

### 2.3 無駄な描画の削減

`fallback-landmass`(タイル未提供のズームレベル用の簡易地図)に
`maxzoom: 5` を追加。本物のベクトルタイルの下に完全に隠れて見えなく
なった後も描画コストを払い続けていた無駄を削減した。

### 2.4 データ精度とファイルサイズの両立

Lアラートの市区町村表示で、当初使っていた簡略化データ(0.1%簡略化、
445KB)は小さな区の形が崩れて認識しづらかったため、より精度の高い版
(1%簡略化、3.8MB)に差し替え。ファイルサイズ増加分はgzip圧縮
(転送量 約1.2MB)とサーバー側の`compression`ミドルウェア導入で吸収した。

## 3. 品質検証: 52パターンのテストケースで発見・修正した不具合

`test/replay_test_cases.py` で、地震・津波・Jアラート・Lアラート(楕円/
市区町村)・気象警報・南海トラフ・火山・降灰・洪水・訓練放送・
デコードエラー等、**52パターン**のテストケースを作成し、ローカル環境で
連続再生して目視検証した。

### 発見した重大な不具合

**サーバー側で、取消・解除信号を1件受信するたびに、保持していた
「現在アクティブな通報」を全件消去していた**(`activeReports = []`)。

これにより、例えば緊急地震速報の取消が1件届いただけで、同時に
アクティブだった無関係の気象警報やLアラートまで、新規接続した
ブラウザから見て消えてしまう状態だった。テストケースを流したところ、
52件中ほぼ全てが最終的に1件しか画面に残らないという明らかな
異常で発覚した。

**修正**: 取消・解除の対象を「災害種別+対象地域」等で判定する
グルーピングキーを実装し、一致するものだけを取り除くよう変更。
判定できない場合は何も消さない(安全側に倒す)設計にした。

修正後、52件中カード表示すべき33件(取消・解除された19件を除く)が
正しく残ることを確認。

### 副次的に発見・修正した不具合

- **Lアラートの解除(All Clear)がサーバー側の「終了信号」判定に
  含まれておらず**、解除後も新規接続したブラウザに再送され続けていた
- **Jアラートに解除処理が一切実装されておらず**、解除メッセージが
  通常のアラートとして積み上がってしまう構成だった
  (Lアラートと同様の仕組みを追加して解消)

## 4. まとめ

| 項目 | 内容 |
|---|---|
| 送信の受信ループブロック時間 | 約150-230ms → 約0.2ms |
| 実送信時間 | 約40%短縮(keep-alive) |
| 送信失敗時の耐性 | 自動リトライ(最大5回)を新規実装 |
| 地図の優先描画 | 警報エリアを最優先、低頻度データは背景読み込み |
| 発見・修正した重大バグ | 1件(取消時の全消去バグ) |
| テストケース数 | 52パターン(自動再生ツール付き) |

いずれも実測・実際のテストデータに基づく改善であり、体感ではなく
数値で裏付けられている。
